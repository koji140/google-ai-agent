# kinsetsu-process Google Workspace再現PoC

## 目的

このPoCは、汎用的なDrive同期や文書台帳の技術検証ではない。

目的は、`kinsetsu-process` リポジトリで管理している近接施工協議の業務設計、判断ログ、手順書、タスク、説明資料を、Google Workspace上で再現し、大野さん・潮田さんがGitHubを直接見なくても業務に参加できる状態を検証することである。

つまり、Claude CodeやCodexに任せているような「考える・整理する・実行する・記録する」体験を、近接施工協議の実業務に寄せてGoogle Workspace上へ移す。

## 方針

- 抽象的な `google-ai-agent-poc` は作らない
- 最初のPoC対象を `kinsetsu-process` のWorkspace再現にする
- ただし本番Drive、本番主DB、実個人情報は直接使わない
- サンプル案件と複製文書で安全に検証する
- 大野さん・潮田さんが使える作業入口をPoC初期から含める
- Google Workspace上の運用が固まってから、本番環境への移行可否を判断する

## 推奨Driveルート

```text
kinsetsu-workspace-agent-poc/
├── 00_業務ハブ/
├── 01_正本文書/
├── 02_案件管理DB/
├── 03_議事録_打合せ/
├── 04_判断ログ/
├── 05_手順書_テンプレート/
├── 06_大野さん_潮田さん作業/
├── 07_検証用サンプル案件/
└── 99_システム_ログ/
```

## フォルダの役割

| フォルダ | 役割 |
|---|---|
| `00_業務ハブ` | 利用者が最初に開く入口。全体説明、リンク集、今週やることを置く |
| `01_正本文書` | `kinsetsu-process` の正本級文書をGoogle Docs化して置く |
| `02_案件管理DB` | サンプル主DB、文書台帳、ToDo台帳、決定事項台帳を置く |
| `03_議事録_打合せ` | 本社打ち合わせ、局ヒアリング、内部打合せの議事録を置く |
| `04_判断ログ` | 方針判断、承認、Go/NoGo、設計変更理由を置く |
| `05_手順書_テンプレート` | 大野さん・潮田さんに渡す作業手順、メール文面、チェックリストを置く |
| `06_大野さん_潮田さん作業` | 2人が実際に見る作業入口、担当別チェックリスト、提出先を置く |
| `07_検証用サンプル案件` | 実個人情報を含まないダミー案件、添付資料、回付メール例を置く |
| `99_システム_ログ` | Apps Script実行ログ、エラーログ、AI抽出ログを置く |

## 大野さん・潮田さん向け作業入口

`06_大野さん_潮田さん作業` には、最初から担当者別の入口を置く。

```text
06_大野さん_潮田さん作業/
├── 大野さん_週次集計/
├── 大野さん_アンケート送付回収/
├── 潮田さん_動作確認チェック/
└── 潮田さん_手順書整備/
```

各フォルダには以下を置く。

- 作業手順書
- 入力・確認対象へのリンク
- 完了条件
- 迷ったときの確認先
- 作業結果を記録するSheetまたはDocs

## 文書台帳で管理する観点

汎用文書台帳ではなく、近接施工協議の業務移植を評価できる列を持たせる。

| 観点 | 例 |
|---|---|
| 元repoパス | `docs/process-design.md` |
| Workspace配置先 | `01_正本文書` |
| 正本区分 | 正本 / 派生 / 作業用 / サンプル / ログ |
| 業務フェーズ | Phase 1 / Phase 2 / 共通 / PoC |
| 利用者 | 石丸 / 大野さん / 潮田さん / アソシ / 局 / 本社 |
| 自律度 | L0からL4 |
| 承認状態 | 下書き / 確認待ち / 承認済み / 実行済み / 廃止 |

## PoCの成功条件

1. `kinsetsu-process` の主要な正本文書がGoogle Workspace上で迷わず参照できる
2. 文書台帳から、元repoパス、Drive URL、正本区分、担当者、承認状態を確認できる
3. 大野さん・潮田さん向けの作業入口があり、作業手順と完了条件が明確になっている
4. サンプル案件で、案件管理DB、手順書、判断ログ、議事録がつながる
5. Apps ScriptでDrive配下の文書を台帳化できる
6. 本番移行前に、何を自動化してよいか、何を承認付きにするか判断できる

## 本番環境へ進む条件

- PoCで扱うサンプル案件の流れが破綻しない
- 文書台帳と主DBの役割分担が明確である
- 大野さん・潮田さんが見るべき入口が1つにまとまっている
- 個人情報、権限、誤送信に関する禁止領域が明文化されている
- 本番主DBや本番Driveを直接変更しないdry runの仕組みがある
