# Google Workspace エージェントアーキテクチャ

## 設計思想

本アーキテクチャの根本的な考え方は、**AIそのものを主役にしない**ことだ。

AIは「賢い相談相手」ではなく、業務フローの中で特定の処理を担当する「部品」として位置づける。入力元、処理内容、出力先、保存先がすべて事前に定義されており、ユーザーが毎回AIに指示を出す必要がない状態を目指す。

## データフロー

### フロー1: 文書登録の自動化

```
Driveにファイル配置 → Apps Scriptがトリガー検知 → ファイル情報を取得
→ Sheets台帳に自動登録（ファイル名、日付、フォルダ、URL、ステータス）
```

このフローは Gemini を使わない。ファイルメタデータだけで完結する最もシンプルなパターン。

### フロー2: 議事録からの自動抽出

```
議事録DocをDriveに保存 → Apps Scriptがトリガー検知
→ Docの本文を取得 → Gemini APIに送信
→ ToDo / 決定事項 / 要約を抽出
→ それぞれSheetsの対応する台帳に自動登録
```

このフローがAI活用の中核。議事録のフォーマットをテンプレートで固定することで、Geminiへのプロンプトも固定でき、抽出精度が安定する。

### フロー3: 要約・分類の自動付与

```
台帳に新規登録あり → Apps Scriptが定期実行
→ まだ要約がない行を検出 → 対象ファイルの本文を取得
→ Gemini APIで要約・分類を生成
→ Sheets台帳の対応列に自動記入
```

### フロー4: レポート自動生成

```
週次トリガー → Apps Scriptが台帳データを集計
→ Gemini APIでサマリ文を生成
→ Google Docsにレポートとして出力
→ 関係者に通知
```

## 「業務フロー部品」としてのAI設計

本アーキテクチャでは、AIは以下の制約の中で動く。

| 制約 | 内容 |
|------|------|
| 入力元が固定 | 特定のDriveフォルダ、特定の台帳Sheet |
| プロンプトが固定 | テンプレート化されたプロンプト（毎回書かない） |
| 出力形式が固定 | JSON or 構造化テキスト → Sheetsの行やDocsのセクション |
| 保存先が固定 | 特定のSheetの特定の列、特定のDocの特定のセクション |
| トリガーが固定 | ファイル作成、定時実行、台帳更新 |

この制約により、以下が実現される。

- **ユーザーはプロンプトを書かない**: AI処理は自動で起動する
- **出力のばらつきが小さい**: 入力と出力の形式が固定されている
- **結果が業務フローにつながる**: 台帳やDocsに自動保存される
- **運用が属人化しない**: 仕組みとして動くので、特定の人に依存しない

## Apps Scriptの役割

Apps Scriptは本アーキテクチャの「制御層」であり、以下を担当する。

1. **トリガー管理**: ファイル作成検知、定時実行、編集検知
2. **データ取得**: Drive内のファイル一覧取得、Docの本文取得
3. **Gemini API呼び出し**: プロンプト送信と結果受信
4. **台帳更新**: Sheetsへの行追加・列更新
5. **Doc生成**: レポート・サマリの自動生成
6. **通知**: メール or チャットでの通知送信

Apps Script自体にAIの機能はない。あくまで「業務フローをつなぐ接着剤」としての役割を果たす。

## Geminiの使い方

Geminiは以下のパターンで使用する。

| 処理 | 入力 | 出力 | プロンプト例（概要） |
|------|------|------|---------------------|
| 要約 | 文書の全文 | 3行要約 | 「以下の文書を3行で要約してください」 |
| ToDo抽出 | 議事録の全文 | ToDo一覧（担当者・期限付き） | 「以下の議事録からToDoを抽出し、担当者と期限を付けてください」 |
| 決定事項抽出 | 議事録の全文 | 決定事項一覧 | 「以下の議事録から決定事項を抽出してください」 |
| 分類 | 文書の全文 | カテゴリタグ | 「以下の文書を[カテゴリ一覧]から分類してください」 |
| 判断ログ生成 | 議事録+背景情報 | 判断理由の構造化記録 | 「以下の内容から、何が判断され、その根拠は何かを整理してください」 |

**重要な設計方針**: Geminiの出力はそのまま使うのではなく、Apps Scriptでパースして構造化データに変換してからSheetsに格納する。Geminiの出力形式が多少ぶれても、パース処理で吸収する。

## セキュリティと権限

- Apps Scriptの実行権限はGoogleアカウントに紐づく
- Drive / Sheets / Docs のアクセス権限はGoogle標準の共有設定で管理
- Gemini APIキーはApps Scriptのプロパティサービスに格納（コードに直書きしない）
- 処理対象のフォルダ・ファイルはスクリプト内で明示的に指定する

## この設計が解くこと・解かないこと

### 解くこと
- 文書の台帳登録の自動化
- 議事録からの情報抽出の自動化
- 要約・分類の自動付与
- 台帳の鮮度維持
- 非エンジニアでも恩恵を受けられるAI活用

### 解かないこと（この設計の外にあること）
- コードの差分管理（→ GitHubの方が適切）
- 高度な検索・全文検索（→ Drive検索の限界がある）
- リアルタイム通知のカスタマイズ（→ Apps Scriptのトリガー間隔に制約あり）
- 大量データの高速処理（→ Sheetsの行数制限、Apps Scriptの実行時間制限あり）
