# 自律度レベル設計

## 目的

業務エージェントをいきなり全自動にすると、誤登録、誤通知、誤判断のリスクが大きい。

そのため、処理ごとに自律度レベルを定義し、どこまでAIとApps Scriptに任せてよいかを明確にする。

## レベル一覧

| レベル | 名前 | 内容 | 人間の関与 |
|---|---|---|---|
| L0 | 手動補助 | AIは使わず、人間が台帳に入力する | 全工程で必要 |
| L1 | 自動下書き | AIやScriptが候補を作るが、実行はしない | 確認と転記が必要 |
| L2 | 承認付き実行 | 候補を作り、承認されたものだけ実行する | 承認が必要 |
| L3 | 条件付き自動実行 | 低リスク処理は自動実行し、例外だけ人間に回す | 例外時に必要 |
| L4 | 高度自律実行 | 複数工程を自動で進め、監査ログだけ残す | 事後確認中心 |

## 初期PoCの推奨

Phase 1では L2 までを上限にする。

- 文書メタデータ登録: L3でもよい
- 種別の推定: L1
- 要約生成: L1
- ToDo登録: L2
- 決定事項登録: L2
- 通知送信: L2
- 正本フラグ更新: L2
- ファイル移動や削除: L0または対象外

## レベル別の実装方針

### L0: 手動補助

自動処理しない。台帳テンプレートや入力フォームだけ用意する。

向いている処理:

- ファイル削除
- 正本の最終確定
- 対外送信前の最終確認

### L1: 自動下書き

AIやScriptは候補を作るだけで、ステータスは `確認待ち` にする。

向いている処理:

- 要約候補
- 文書種別候補
- ToDo候補
- 決定事項候補
- 判断ログ候補

### L2: 承認付き実行

人間が `承認済み` に変更した行だけ実行する。

向いている処理:

- 担当者への通知
- ToDo台帳への正式登録
- 決定事項台帳への正式登録
- 週次サマリの送信

### L3: 条件付き自動実行

ルールが明確で、失敗しても復旧しやすい処理だけ自動実行する。

向いている処理:

- Driveファイルの台帳登録
- 実行ログ追記
- 期限が近いToDoのリマインド
- 未分類ファイルの検出

### L4: 高度自律実行

初期段階では採用しない。

複数の台帳更新、外部通知、文書生成、関係者連絡を自動でつなぐ段階。運用実績ができてから対象を限定して検討する。

## 自律実行の禁止領域

初期PoCでは以下を自動実行しない。

- Driveファイルの削除
- 正本ファイルの上書き
- 外部関係者へのメール送信
- 法務・契約・金額に関わる確定判断
- 人事評価や個人に不利益が出る判断
- 権限変更

## 台帳に必要な列

自律度を管理するため、主要台帳には以下の列を持たせる。

| 列 | 目的 |
|---|---|
| 自律度 | L0からL4 |
| ステータス | 下書き、確認待ち、承認済み、実行済みなど |
| 承認者 | 実行を許可した人 |
| 承認日時 | 承認された日時 |
| 実行可否 | 自動実行対象かどうか |
| 実行結果 | 成功、失敗、スキップ |
| 判断理由 | AIまたは人間の判断理由 |
