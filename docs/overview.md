# 全体像（Overview）

## このプロジェクトが解こうとしている問題

多くの企業がAIの業務活用を試みているが、「ChatGPTに聞く」「Geminiで要約する」といった使い方は定着しにくい。理由は明確で、汎用AIの利用には毎回プロンプトを考える必要があり、出力はチャット画面の中で消え、業務フローの次の工程につながらないからだ。

AIを「便利なツール」として個人の裁量に委ねる限り、使う人と使わない人の差が開き、組織としての定着には至らない。

## 本プロジェクトの着眼点

この問題を解くために、本プロジェクトでは以下の転換を行う。

| 従来の考え方 | 本プロジェクトの考え方 |
|---|---|
| AIを相談相手として使う | AIを業務フローの部品として組み込む |
| 毎回プロンプトを考える | 入力・出力・保存先を固定する |
| 出力はチャットで終わる | 出力は台帳・ドキュメントに自動保存される |
| 個人の裁量で使う | 承認境界を決めた上で業務プロセスの一部として実行される |
| ツールの習熟が必要 | 既存のGoogle製品の延長で動く |

## なぜ Google Workspace なのか

本プロジェクトは、基盤をGoogle Workspace製品（Drive / Sheets / Docs / Apps Script / Gemini）に限定する。

この判断の根拠は以下の通り。

1. **既に導入済みの企業が多い**: 新しいツールを導入する承認プロセスが不要
2. **非エンジニアが使える**: Sheets や Docs は業務で日常的に使われている
3. **自動化の仕組みが内蔵されている**: Apps Script でトリガー・API連携が可能
4. **AI機能が統合されつつある**: Gemini がWorkspace全体に組み込まれる方向にある
5. **共有・権限管理が標準装備**: チーム運用に必要な仕組みが最初からある

GitHub + CLI + Python のような技術スタックは強力だが、「現場の担当者が自分で運用できるか」という観点では障壁が高い。Google Workspace方式は、その障壁を大幅に下げる。

## どんな業務課題を解くのか

具体的には、以下のような業務課題を対象とする。

- **文書管理**: 打ち合わせ資料、議事録、設計書、HTMLファイルなどが散在し、何がどこにあるか分からない
- **判断の記録**: 「なぜこの判断をしたか」が残らず、後から検証できない
- **ToDo管理**: 会議で決まったことが議事録の中に埋もれ、実行されない
- **台帳維持**: 手動で台帳を更新する運用は、遅かれ早かれ崩壊する
- **要約・分類**: 大量の文書を人手で読み、分類する時間がない
- **実行の属人化**: 次に何をすべきか分かっていても、担当者への通知、台帳更新、記録が人手に依存する

これらはいずれも「情報はあるが、整理・構造化・次工程への接続が弱い」という共通の構造を持つ。

## 目指す状態

本プロジェクトが目指す状態は、以下のようなものだ。

1. **Driveに文書を置くだけで台帳に登録される**
2. **議事録を書くだけでToDo・決定事項が抽出される**
3. **台帳が常に最新に保たれる**
4. **要約・分類がAIによって自動付与される**
5. **AIが判断候補を作り、人間が承認したものだけ実行される**
6. **実行結果と判断理由がログとして残る**
7. **ユーザーは「AI」を意識せず、普段の業務フローの中で恩恵を受ける**

この状態を、Google Workspace製品の組み合わせだけで実現することが、本プロジェクトの目標である。

## 現在のフェーズ

本プロジェクトは**初期PoCフェーズ**にある。

- Google Workspaceだけで業務エージェントを動かす方針を整理している
- Phase 1の最小Apps Script実装を用意している
- 事実と仮説を区別しながら、実運用に必要な承認、ログ、制約を設計している
- 次のステップとして、Google Drive / Sheets / Apps Script上でPoC環境を作る
