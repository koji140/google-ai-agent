# 業務エージェントの運用モデル

## 目的

業務エージェントは、単にAIで文章を生成する仕組みではない。

入力を受け取り、必要な情報を読み、判断候補を作り、人間の承認を挟みながら実行し、結果を記録する一連の運用モデルとして設計する。

## 基本サイクル

```text
入力 -> 読解 -> 分解 -> 判断候補 -> 承認 -> 実行 -> 記録 -> 振り返り
```

## 1. 入力

業務イベントを起点にする。

| 入力元 | 例 |
|---|---|
| Drive | 資料、議事録、HTML、成果物、判断ログ |
| Gmail | 依頼、送付資料、問い合わせ、通知 |
| Calendar | 会議予定、期限、定例イベント |
| Forms | 判断ログ、手動登録、作業依頼 |
| Chat | 確認依頼、承認依頼、簡易指示 |

## 2. 読解

Apps Scriptが対象ファイルやメッセージを取得し、Geminiに渡せるテキストに整える。

- Google Docs本文の取得
- PDFや添付ファイルの扱い方の整理
- ファイル名、フォルダ名、更新日時、作成者などのメタデータ取得
- 議事録テンプレートのセクション分解

## 3. 分解

AIの出力は、そのまま文章として保存せず、台帳に入る単位へ分解する。

| 分解先 | 内容 |
|---|---|
| 文書台帳 | ファイルID、種別、URL、要約、ステータス |
| ToDo台帳 | タスク、担当者、期限、出典、状態 |
| 決定事項台帳 | 決定内容、決定日、根拠、関連議題 |
| 判断ログ台帳 | 判断内容、理由、影響範囲、承認者 |
| 実行ログ | 実行日時、処理名、対象、結果、エラー |

## 4. 判断候補

エージェントは最終判断者ではなく、判断候補を作る。

例:

- この資料は「議事録」と推定される
- この文にはToDoが含まれる可能性が高い
- この決定事項は関係者承認が必要
- このファイルは正本候補だが、既存正本と競合している

## 5. 承認

業務影響がある処理は、Sheets上のステータスやChat/Gmailの承認を挟む。

承認状態の例:

| 状態 | 意味 |
|---|---|
| 下書き | AIまたは自動処理が作成しただけ |
| 確認待ち | 人間の確認が必要 |
| 承認済み | 後続処理を実行してよい |
| 差戻し | 修正が必要 |
| 実行済み | 承認後の処理が完了 |
| 廃止 | 無効化された記録 |

## 6. 実行

承認済みの行だけをApps Scriptが処理する。

実行例:

- 文書台帳のステータス更新
- ToDo担当者への通知
- 期限リマインド
- 週次サマリDocsの生成
- 正本フラグの更新
- 関係者へのメール送信

## 7. 記録

自動処理は必ず記録する。

最低限残す項目:

- 実行日時
- 実行関数
- 対象ID
- 入力元
- 実行結果
- エラー内容
- AIが出した判断理由
- 人間の承認者

## 8. 振り返り

台帳の履歴から、週次や月次で運用を改善する。

- どの種類の文書が増えているか
- 未完了ToDoがどこに滞留しているか
- AI抽出の修正が多い項目は何か
- 承認待ちが長い処理は何か
- エラーが多いフォルダや文書形式は何か
