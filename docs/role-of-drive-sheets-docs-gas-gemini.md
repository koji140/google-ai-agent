# 各コンポーネントの役割

## 概要

本アーキテクチャは5つのGoogle製品を組み合わせて構成される。それぞれの製品には明確な役割があり、得意なこと・不得意なことがある。この文書では、各コンポーネントの責務を整理し、どう組み合わせるかを定義する。

---

## Google Drive — 文書庫

### 役割
すべての業務文書の保管場所。ファイルの「置き場所」であり、フォルダ構造が文書の分類体系を兼ねる。

### 責務
- 打ち合わせ資料、議事録、HTML、成果物、テンプレートの保管
- フォルダ構造による文書分類
- アクセス権限の管理
- ファイルの共有・公開設定
- 版管理（Google Docsの場合は変更履歴が自動保存）

### 向いていること
- ファイルの集約管理
- チームでの共有
- フォルダベースの分類
- Google Docs/Sheets との連携
- モバイルからのアクセス

### 向いていないこと
- 構造化データの管理（→ Sheetsの役割）
- テキストベースの差分管理（→ Gitの方が得意）
- 複雑な検索条件（Drive検索はあるが、全文検索の精度には限界がある）
- HTMLやコードの編集・プレビュー

### フォルダ構造の設計例

```
プロジェクトルート/
├── 01_打ち合わせ資料/
│   ├── 2024-01/
│   └── 2024-02/
├── 02_議事録/
├── 03_HTML/
├── 04_成果物/
├── 05_判断ログ/
├── 06_テンプレート/
└── 99_台帳/（Sheetsを格納）
```

## Google Sheets — 台帳DB

### 役割
構造化データの管理。文書台帳、ToDo台帳、決定事項台帳、判断ログ台帳など、「一覧で見たい情報」を管理する。

### 責務
- 文書メタデータの台帳管理（ファイル名、日付、分類、URL、ステータス）
- ToDo一覧の管理（タスク、担当者、期限、ステータス）
- 決定事項一覧の管理（日付、内容、根拠、関連会議）
- 判断ログの蓄積（日付、判断内容、理由、関連文書）
- フィルタ・ソート・ピボットによるデータ分析

### 向いていること
- 一覧性のあるデータ管理
- フィルタリング・ソート
- 数式による集計・計算
- Apps Scriptからの読み書き
- 複数人での同時編集
- データのエクスポート（CSV等）

### 向いていないこと
- 大量データ（数万行を超えると動作が重くなる）
- リレーショナルデータ（外部キーや結合はない）
- 長文テキストの管理（→ Docsの役割）
- 複雑な権限管理（シート単位の権限はあるが、行単位の権限はない）

### 台帳設計の例

**文書台帳**

| 列 | 内容 |
|----|------|
| A: ファイル名 | ファイル名 |
| B: 種別 | 打ち合わせ資料 / 議事録 / HTML / 成果物 / 判断ログ |
| C: 作成日 | yyyy-mm-dd |
| D: 更新日 | yyyy-mm-dd |
| E: フォルダ | 格納先フォルダパス |
| F: URL | DriveのファイルURL |
| G: 要約 | Geminiによる自動要約（3行以内） |
| H: ステータス | 作成中 / 確定 / 廃止 |
| I: 登録方法 | 自動 / 手動 |

## Google Docs — 長文ドキュメント / ハブ

### 役割
長文の文書作成と、情報のハブ。議事録、設計書、レポート、判断経緯の記録など、文脈を持った文書を扱う。

### 責務
- 議事録の作成・管理
- 設計書・仕様書の作成
- 週次レポート・サマリの出力先
- テンプレートの元ファイル
- 判断経緯のナラティブ記録

### 向いていること
- 文脈のある長文の作成・編集
- テンプレートからの複製
- 変更履歴の自動記録
- コメント・提案機能による共同編集
- Apps Scriptからの本文読み取り・書き込み

### 向いていないこと
- 構造化データの管理（→ Sheetsの役割）
- HTMLのレンダリング・プレビュー
- 差分の可視化（変更履歴はあるが、Gitのdiffとは異なる）
- プログラムコードの管理

### テンプレート活用

議事録テンプレートの例:

```
# 議事録: [会議名]
日付: [yyyy-mm-dd]
参加者: [名前一覧]
場所: [場所/オンライン]

## 議題
1. ...

## 議論内容
### 議題1: [議題名]
- ...

## 決定事項
- [ ] ...

## ToDo
- [ ] [担当者] [タスク内容] [期限: yyyy-mm-dd]

## 次回予定
- 日時: 
- 議題: 
```

テンプレートを統一することで、Geminiによる抽出精度が大幅に向上する。

## Google Apps Script — 自動化エンジン

### 役割
Drive / Sheets / Docs / Gemini APIをつなぐ「制御層」。業務フローの自動化ロジックを実装する。

### 責務
- Driveの監視（新規ファイル検知、フォルダスキャン）
- Sheetsの読み書き（台帳の自動更新）
- Docsの読み書き（本文取得、セクション挿入）
- Gemini APIの呼び出し
- 時間ベースのトリガー実行
- メール・チャット通知

### 向いていること
- Google製品間の連携
- 定期実行（時間トリガー）
- イベント駆動（編集トリガー）
- プロトタイプの迅速な開発
- 追加インフラ不要

### 向いていないこと
- 長時間の処理（実行時間上限: 6分/回）
- 大量のAPI呼び出し（日次クォータあり）
- 複雑なエラーハンドリング
- デバッグのしやすさ（IDE機能が限定的）
- バージョン管理（→ clasp で Git 連携は可能だが、本プロジェクトの想定外）

### トリガーの種類

| 種類 | 用途 |
|------|------|
| 時間ベース（毎時/毎日/毎週） | 台帳の定期スキャン、レポート生成 |
| ファイル編集時 | Sheets更新時の処理起動 |
| フォーム送信時 | 入力フォームからの台帳登録 |
| カスタムメニュー | 手動実行の入口 |

## Gemini — 要約 / 抽出 / 分類のAI

### 役割
テキストの理解と生成を担当する。人間が読んで判断していた作業を代行する。

### 責務
- 文書の要約生成
- 議事録からのToDo抽出
- 議事録からの決定事項抽出
- 文書の分類・タグ付け
- 判断理由の構造化

### 向いていること
- 自然言語テキストの理解
- 構造化されていない文書からの情報抽出
- 要約の生成
- 分類・ラベリング

### 向いていないこと
- 100%正確な情報抽出（ハルシネーションのリスク）
- 数値計算
- 最新情報の把握（学習データの時点まで）
- 大量文書の高速処理（API呼び出しにはコストと時間がかかる）

### 利用上の注意

- **出力を鵜呑みにしない**: Geminiの出力は「下書き」であり、重要な判断は人間が確認する
- **プロンプトを固定する**: 毎回異なるプロンプトを書くのではなく、用途ごとにテンプレート化する
- **出力形式を指定する**: JSON形式での出力を指定し、Apps Scriptでパースする
- **フォールバックを用意する**: API障害時は台帳への登録だけ行い、AI処理は後で再実行する

---

## 組み合わせ方の基本原則

1. **Drive が入口**: すべての文書はまずDriveに置かれる
2. **Apps Script が検知して動く**: トリガーでイベントを拾い、処理を起動する
3. **Gemini が賢い処理を担当**: テキスト理解が必要な部分だけGeminiに渡す
4. **Sheets に結果を蓄積**: 構造化された結果は台帳に格納する
5. **Docs に文脈を残す**: 長文のレポートや判断経緯はDocsに出力する

この流れを「業務パイプライン」として定義し、各パイプラインを再利用可能にすることが設計目標である。
