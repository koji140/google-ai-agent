# 実装ロードマップ

## 基本方針

いきなり全部を自動化しない。まず最もシンプルな自動化から始め、動くものを確認しながら段階的に拡張する。

各フェーズの設計方針:
- **小さく始める**: 一つのフォルダ、一つの台帳から
- **手動を残す**: 最初は手動運用と並行し、自動化の精度を検証する
- **壊れにくくする**: 自動化が止まっても業務が止まらないようにする
- **検証してから広げる**: 精度と実用性が確認できてから対象を拡大する
- **承認境界を明確にする**: AIの判断候補と自動実行を分ける
- **必ず記録する**: 実行結果、エラー、判断理由を台帳に残す

## Phase 1: kinsetsu-process再現用文書台帳の自動更新

### 目的
`kinsetsu-process` の正本文書、判断ログ、手順書、担当者作業フォルダをGoogle Workspace上に再現し、Driveに文書を置くだけでSheetsの文書台帳に自動登録される仕組みを作る。

### 対象
- `kinsetsu-workspace-agent-poc` Driveフォルダ
- Google Sheets 1つ（文書台帳）
- `kinsetsu-process` から移植する正本文書・手順書・サンプル案件

### 成果物
- 文書台帳Sheet（テンプレート）
- Apps Script（Driveフォルダスキャン → Sheets登録）
- 時間トリガー設定（例: 1時間ごと）
- 実行ログSheet
- エラーログSheet

### 実装内容
1. 近接施工Workspace再現用の文書台帳Sheet構造を定義
2. 対象フォルダのファイル一覧を取得するApps Scriptを作成
3. 新規ファイルを検知し、台帳に行を追加するロジックを実装
4. 時間トリガーを設定（まずは1時間間隔）
5. 重複登録を防ぐ仕組み（ファイルIDで一意性を確保）
6. 実行ログ・エラーログを記録する
7. dry runモードで登録予定の内容を確認できるようにする
8. フォルダ名から、正本区分、業務フェーズ、主な利用者を初期推定する

### 成功条件
- フォルダにファイルを置いたら、1時間以内に台帳に登録される
- 手動で台帳を更新する必要がなくなる
- 既存ファイルも初回スキャンで登録される
- 失敗時にエラー内容を後から確認できる
- 自動実行前にdry runで差分を確認できる
- 大野さん・潮田さん向けの作業フォルダが台帳上で識別できる
- 元repoパスとWorkspace配置先の対応を追跡できる

### Gemini不要
このフェーズではAIは使わない。ファイルメタデータの自動登録、ログ、承認境界の土台作りに集中する。

### 想定期間
1〜2週間（台帳設計 + スクリプト開発 + テスト）

## Phase 2: 議事録からToDo・決定事項の自動抽出

### 目的
議事録Docから、ToDo・決定事項・要約をGeminiで自動抽出し、各台帳に登録する。

### 前提
- Phase 1 が動いていること
- 議事録テンプレート（Google Docs）が定義されていること

### 対象
- 議事録フォルダ内のGoogle Docs
- ToDo台帳Sheet
- 決定事項台帳Sheet
- 文書台帳の要約列

### 成果物
- 議事録テンプレート（Google Docs）
- ToDo台帳Sheet
- 決定事項台帳Sheet
- Apps Script（議事録解析 → Gemini API → 各台帳登録）
- 承認待ちステータスを持つ抽出台帳

### 実装内容
1. 議事録テンプレートのセクション構造を定義
2. Gemini API呼び出し用のプロンプトテンプレートを作成
3. 議事録本文を取得 → Gemini APIに送信 → 結果をパースするApps Scriptを実装
4. パース結果をToDo台帳・決定事項台帳に「確認待ち」として登録するロジック
5. 要約を文書台帳の要約列に記入するロジック
6. エラーハンドリング（API障害時のリトライ、パース失敗時のログ）
7. 承認済みのToDo・決定事項だけ正式扱いにするステータス遷移

### 成功条件
- 議事録を書いて保存したら、ToDoと決定事項の候補が自動で台帳に登録される
- 抽出精度が実用レベル（8割以上の正確性）
- 手動での確認・修正が容易
- 承認された候補だけが正式なToDo・決定事項になる

### 想定期間
2〜3週間（テンプレート設計 + プロンプト調整 + スクリプト開発 + 精度検証）

## Phase 3: 要約・分類・通知の自動化

### 目的
台帳に登録された文書に対し、自動で要約・分類を付与し、承認済みの内容だけ関係者に通知する。

### 前提
- Phase 1, 2 が動いていること

### 対象
- 文書台帳の全文書
- 通知先（メール or Google Chat）

### 成果物
- 要約・分類の自動付与スクリプト
- 通知スクリプト
- 分類マスタ（カテゴリ一覧）

### 実装内容
1. 文書台帳で「要約が未記入の行」を検出するロジック
2. 対象ファイルの本文を取得 → Gemini APIで要約・分類を生成
3. 結果を台帳に記入
4. 承認済み行に対する通知メール送信
5. 週次サマリの自動生成（Docsへの出力）
6. 通知前のプレビューと送信ログ記録

### 成功条件
- 台帳に登録された文書に要約と分類が自動付与される
- 通知が適切なタイミングで届く
- 週次サマリが自動生成される
- 誰に何を通知したかがログに残る

### 想定期間
2〜3週間

## Phase 4: 運用ルール整備・正本管理フロー確立

### 目的
自動化の仕組みが安定稼働した上で、運用ルールを整備し、チームでの運用を確立する。

### 対象
- Drive内のフォルダ構造・命名規則
- 台帳の運用ルール
- 正本管理のフロー
- 権限設定

### 成果物
- 運用ルールドキュメント（Google Docs）
- フォルダ構造テンプレート
- 命名規則ガイド
- 正本フラグの運用ルール
- チーム向けオンボーディング資料

### 実装内容
1. ファイル命名規則の策定と適用
2. フォルダ構造の標準化
3. 正本フラグ（台帳上の「確定」ステータス）の運用ルール定義
4. アクセス権限の設計と設定
5. 運用マニュアルの作成
6. 障害時の復旧手順

### 成功条件
- チームメンバーが自律的に運用できる
- 新規メンバーがオンボーディング資料を見て運用に参加できる
- 正本の所在が常に明確

### 想定期間
2〜4週間

---

## フェーズ間の依存関係

```
Phase 1（文書台帳）
    ↓ 台帳が動いている前提
Phase 2（議事録抽出）
    ↓ 抽出が動いている前提
Phase 3（要約・分類・通知）
    ↓ 自動化が安定している前提
Phase 4（運用ルール整備）
```

Phase 1 → Phase 2 は強い依存関係がある。Phase 3 と Phase 4 は部分的に並行可能。

## リスクと対策

| リスク | 対策 |
|--------|------|
| Geminiの抽出精度が低い | テンプレートの改善、プロンプトの調整、人間による確認フローの維持 |
| Apps Scriptの実行時間超過 | 処理対象を分割、バッチ処理の設計 |
| 台帳が巨大になる | 月次でアーカイブ、フィルタ表示の活用 |
| 運用ルールが守られない | 自動チェック機能の追加、定期的なルール見直し |
| 自動実行の誤爆 | 自律度レベルを設定し、重要処理は承認付きにする |
| AIの判断理由が追えない | 判断理由、プロンプト種別、出典URLを台帳に残す |
