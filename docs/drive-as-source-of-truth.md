# Drive正本運用の考え方

## 基本方針

本プロジェクトでは、**Google Driveを運用正本の置き場所とする**。

これは「Drive上にあるファイルが、常に最新かつ正式な版である」という運用ルールを意味する。ローカルPCやメール添付のファイルはあくまで「コピー」であり、Drive上のファイルが唯一の正本となる。

## なぜDrive正本なのか

### メリット
1. **単一の参照先**: 「最新版はどれか」という問いに対する答えが常に一つ
2. **リアルタイム共有**: 変更が即座に全員に反映される
3. **権限管理**: 閲覧・編集・コメントの権限をGoogleの仕組みで制御できる
4. **変更履歴**: Google Docsは自動で変更履歴を保存する
5. **Apps Scriptとの連携**: 正本がDriveにあれば、自動化処理が直接アクセスできる
6. **モバイルアクセス**: どこからでも正本にアクセスできる

### 注意点
1. **オフライン環境での作業**: インターネット接続がない環境ではアクセスできない（オフラインモードはあるが限定的）
2. **大容量ファイル**: 動画や大量のCADデータなどはDriveの容量を圧迫する
3. **外部関係者との受け渡し**: Google Workspaceアカウントを持たない相手との共有には工夫が必要

## ローカル作業との役割分担

一部の作業はローカルPCで行う必要がある。その場合の役割分担を明確にする。

| 作業 | 場所 | 理由 |
|------|------|------|
| 文書の閲覧・編集 | Drive（Google Docs/Sheets） | リアルタイム共有、変更履歴 |
| HTML制作 | ローカル → Driveにアップロード | エディタやブラウザプレビューが必要 |
| 画像・図面の編集 | ローカル → Driveにアップロード | 専用ソフトが必要 |
| Apps Script開発 | Apps Scriptエディタ（Web） | Google標準の開発環境 |
| テンプレート設計 | Drive（Google Docs/Sheets） | テンプレートはDrive上で管理 |

### ルール
- ローカルで編集したファイルは、**編集完了後すぐにDriveにアップロードする**
- Driveにアップロードした時点で、そのファイルが正本になる
- ローカルの古いコピーは削除するか、明示的に「ローカルコピー」とわかるようにする
- ローカルでの編集中にDrive側が更新されていないか確認する運用を推奨

## Drive正本型運用の設計要素

### 1. ファイル命名規則

命名規則を統一することで、台帳との突合やソート、検索がしやすくなる。

**推奨フォーマット**:
```
[種別]_[プロジェクト名]_[内容]_[日付].[拡張子]

例:
議事録_近接施工_第3回定例会議_20240415.docx
資料_近接施工_施工計画書_v2_20240420.pdf
HTML_近接施工_工事概要_20240501.html
```

**命名規則の要素**:
| 要素 | 内容 | 例 |
|------|------|-----|
| 種別 | 文書の分類 | 議事録、資料、HTML、判断ログ、ToDo |
| プロジェクト名 | 案件名の短縮形 | 近接施工 |
| 内容 | 文書の具体的な内容 | 第3回定例会議 |
| 日付 | yyyymmdd形式 | 20240415 |
| 版 | 必要な場合のみ | v1, v2 |

### 2. 版管理

Google Docsの場合:
- 変更履歴が自動で保存される
- 重要なマイルストーンでは「名前付きバージョン」を作成する
- 「確定版」「ドラフト」の区別はファイル名またはステータス列で管理

Google Docs以外のファイル（PDF、HTML等）の場合:
- ファイル名に版番号を含める（`_v1`, `_v2`）
- 旧版は`_archive/`フォルダに移動する
- 台帳のステータス列で最新版を管理する

### 3. 正本フラグ

文書台帳（Sheets）に「ステータス」列を設け、以下の値で管理する。

| ステータス | 意味 |
|-----------|------|
| ドラフト | 作成中。内容が確定していない |
| レビュー中 | 確認待ち |
| 確定 | 正式な版として承認済み。正本として扱う |
| 廃止 | 古くなった、または取り下げられた文書 |
| アーカイブ | 参照用に保存。現在の運用には使用しない |

### 4. フォルダ構造

```
プロジェクトルート/
├── 01_打ち合わせ資料/
│   ├── 2024-01/
│   ├── 2024-02/
│   └── _archive/
├── 02_議事録/
│   ├── 2024-01/
│   └── _archive/
├── 03_HTML/
│   ├── current/（最新版のみ）
│   └── _archive/
├── 04_成果物/
├── 05_判断ログ/
├── 06_テンプレート/（議事録テンプレート等）
└── 99_台帳/
    ├── 文書台帳.gsheet
    ├── ToDo台帳.gsheet
    ├── 決定事項台帳.gsheet
    └── 判断ログ台帳.gsheet
```

### 5. 更新手順

文書を更新する際の手順:

1. Drive上の対象ファイルを開く（Google Docs/Sheetsの場合は直接編集）
2. 編集完了後、必要に応じて「名前付きバージョン」を作成
3. 台帳のステータス・更新日を確認（自動更新が設定されていれば不要）
4. 大きな変更の場合は、台帳の備考欄にメモを残す

ローカルファイルを更新する場合:

1. ローカルで編集
2. Driveにアップロード（既存ファイルを置き換え or 新版としてアップロード）
3. 旧版がある場合は`_archive/`に移動
4. 台帳が自動更新されることを確認

## Google製品だけで閉じる場合の運用ルール

本プロジェクトの前提は「Google製品だけで閉じる」こと。したがって、Git等の外部ツールに頼らずに正本管理を行う必要がある。

### Git的な機能の代替

| Git の機能 | Google Workspace での代替 | 限界 |
|-----------|--------------------------|------|
| commit | Google Docsの変更履歴 / 名前付きバージョン | コミットメッセージに相当するものがない |
| diff | Google Docsの「変更内容を表示」 | 行単位の差分比較はできない |
| branch | ファイルのコピー（「ドラフト版」として） | マージの仕組みがない |
| merge | 手動でのコピー＆ペースト | 自動マージは不可能 |
| tag | 名前付きバージョン / ファイル名に版番号 | 軽量 |
| blame | Google Docsの変更履歴で確認 | 大まかにしか分からない |

### 運用上の工夫
- **重要な変更前にはコピーを作る**: 大きな変更を加える前に、ファイルをコピーして`_archive/`に保存する
- **台帳の備考欄を活用する**: 変更内容や理由を台帳の備考列にメモする（commitメッセージの代わり）
- **正本フラグを厳密に管理する**: 「確定」ステータスのファイルのみが正本
- **定期的な棚卸し**: 月次で台帳とDriveの実態を突合し、齟齬がないか確認する

## まとめ

Drive正本運用は、Google Workspace内で「唯一の真実の情報源」を維持するための仕組みである。Gitの差分管理ほどの精密さはないが、命名規則・フォルダ構造・ステータス管理・台帳連携を組み合わせることで、実用的な正本管理が実現可能と考える。

ただし、この運用は**ルールの遵守**に大きく依存する。ルールが守られなくなると、正本がどこにあるか分からなくなるリスクがある。Phase 4の運用ルール整備で、このリスクを最小化する設計を行う。
