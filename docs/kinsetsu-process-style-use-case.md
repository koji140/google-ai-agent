# 近接施工協議スタイルのユースケース

## 背景

近接施工協議のような業務は、以下の特徴を持つ。

- **文書量が多い**: 打ち合わせ資料、議事録、施工計画書、協議書、HTML成果物が大量に発生する
- **判断の記録が重要**: 「なぜこの工法を選んだか」「どの条件で合意したか」が後から検証される
- **関係者が多い**: 発注者、施工者、設計者、第三者機関など複数の組織が関わる
- **工程が長い**: 数ヶ月〜数年にわたるプロジェクトで、過去の判断を参照する必要がある
- **非エンジニアが中心**: 建設業の現場担当者はGitやCLIに馴染みがない

このような業務を、GitHub方式（kinsetsu-processリポジトリのようなアプローチ）で設計した状態から、Google Workspace製品だけでどこまで業務運用として再現できるかを検討する。

最初のPoCでは、抽象的なフォルダ構成ではなく `kinsetsu-workspace-agent-poc` を作り、`kinsetsu-process` の正本文書、案件管理DB、判断ログ、手順書、大野さん・潮田さん向け作業入口を再現対象にする。

## 管理対象と対応方針

### 1. 打ち合わせ資料

**従来**: 紙 or ファイルサーバー or メール添付で散在

**Google Workspace方式**:
- Drive内の `03_議事録_打合せ/` または `01_正本文書/` に格納
- 資料をアップロードした時点で、Apps Scriptが検知し文書台帳（Sheets）に自動登録
- 登録情報: ファイル名、日付、種別、URL、アップロード者
- Geminiによる要約を台帳に自動付与（仮説: 数行の要約で十分）

**実現可能性**: 高い。Drive + Sheets + Apps Script で完結する。

### 2. 議事録

**従来**: Word/メモで作成 → メールで共有 → 各自がローカルに保存

**Google Workspace方式**:
- Google Docsのテンプレートから議事録を作成
- テンプレートには「議題」「議論内容」「決定事項」「ToDo」のセクションを固定
- 議事録完成後、Apps Scriptが本文を取得し、Gemini APIに送信
- Geminiが「ToDo」「決定事項」「要約」を抽出
- 抽出結果を各台帳（ToDo台帳Sheet、決定事項台帳Sheet）に自動登録
- 文書台帳にも議事録として登録

**実現可能性**: 高い。ただし、テンプレートの遵守率がGeminiの抽出精度に直結する。

### 3. HTML

**従来**: GitHub管理 or ファイルサーバー管理

**Google Workspace方式**:
- Drive内の `07_検証用サンプル案件/` または関連する説明資料フォルダに格納
- HTMLファイルの直接編集はGoogle Workspace内では困難
- ただし、HTMLファイルの台帳管理（メタデータ管理）は可能
- 更新検知 → 台帳更新 → 変更サマリ生成（Geminiでdiffの要約）は実装可能

**実現可能性**: 中程度。台帳管理は可能だが、HTML自体の編集・プレビュー・差分管理はGoogle Workspaceの弱点。HTMLの制作自体は外部ツール（VSCode等）で行い、成果物をDriveに格納する運用になる。

**補足**: これはGoogle Workspace方式の明確な限界の一つ。HTMLやコードの管理が必要な業務では、GitHub方式との併用を検討すべき。

### 4. 判断ログ

**従来**: 議事録の中に埋もれている / 記録されない

**Google Workspace方式**:
- 判断ログ台帳（Sheets）を用意
- 議事録からGeminiが判断事項を自動抽出
- 台帳に登録: 日付、判断内容、判断理由、関連会議、関連文書URL
- 手動入力の口もGoogle Formsで用意（会議外の判断も記録できる）
- 蓄積された判断ログをGeminiで横断検索・要約することも将来的に可能

**実現可能性**: 高い。議事録テンプレートで「判断事項」セクションを設ければ、抽出精度はさらに上がる。

### 5. ToDo

**従来**: 議事録に書かれるが追跡されない / 個人のメモに分散

**Google Workspace方式**:
- ToDo台帳（Sheets）に一元管理
- 議事録からGeminiが自動抽出: タスク内容、担当者、期限
- ステータス列: 未着手 / 進行中 / 完了 / 保留
- Apps Scriptで期限前リマインド通知
- フィルタ表示で「自分のToDo」「今週期限のToDo」を確認

**実現可能性**: 高い。ただし、担当者名のゆれ（表記の不統一）がGeminiの抽出精度に影響するため、担当者マスタとの突合が望ましい。

### 6. 決定事項

**従来**: 議事録の中に記載されるが、横断的な一覧がない

**Google Workspace方式**:
- 決定事項台帳（Sheets）に一元管理
- 議事録からGeminiが自動抽出: 決定内容、決定日、関連議題、根拠
- 台帳で横断検索が可能
- 将来的に「類似の決定事項」をGeminiで検索し、判断の一貫性を確認する機能も考えられる

**実現可能性**: 高い。

## 全体の管理フロー

```
打ち合わせ
    ↓
議事録作成（Docs テンプレート）
    ↓
Drive に保存
    ↓
Apps Script が検知
    ├── 文書台帳に登録（Sheets）
    ├── Gemini で議事録解析
    │   ├── ToDo抽出 → ToDo台帳（Sheets）
    │   ├── 決定事項抽出 → 決定事項台帳（Sheets）
    │   ├── 判断ログ抽出 → 判断ログ台帳（Sheets）
    │   └── 要約生成 → 文書台帳の要約列
    └── 通知（メール / Chat）
```

## 台帳構成の全体像

| 台帳 | 主な列 | 更新方法 |
|------|--------|---------|
| 文書台帳 | ファイル名、種別、日付、URL、要約、ステータス | 自動（Drive監視） |
| ToDo台帳 | タスク、担当者、期限、ステータス、出典（議事録URL） | 自動（Gemini抽出） + 手動 |
| 決定事項台帳 | 決定内容、日付、根拠、関連議題、出典 | 自動（Gemini抽出） |
| 判断ログ台帳 | 判断内容、理由、日付、関連文書 | 自動（Gemini抽出） + 手動 |

## 擬似レポとしてどこまで再現できるか

### 再現できること
- 文書の一元管理と台帳化
- 議事録からの自動情報抽出
- ToDo・決定事項の追跡
- 判断ログの蓄積
- 要約・分類の自動付与
- チーム内での共有・共同作業

### 再現が難しいこと
- HTMLやコードの差分管理
- ブランチ・マージによる並行作業管理
- Pull Requestに相当するレビューフロー
- CI/CDによる自動ビルド・デプロイ
- 大量のファイルの一括処理

### 対応方針
再現が難しい領域については、無理にGoogle Workspace内で解決しようとせず、「Google Workspace方式でカバーする範囲」と「それ以外」を明確に分ける。本プロジェクトの初期フェーズでは、再現できる部分に集中する。
