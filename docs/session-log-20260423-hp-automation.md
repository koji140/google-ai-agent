# Session Log: 2026-04-23 ホームページ改訂ワークフローの自動化

## 1. プロジェクトの背景と目的
大野さん（非エンジニア）が、GitHub やコマンド操作に触れることなく、Google Drive と Gemini だけで「近接施工ホームページ」を安全に改訂し続けられる環境を構築する。

## 2. 思考プロセスとアーキテクチャの変遷

### フェーズ1：HTMLの部品化（成功）
- **課題**: 1枚の長いHTMLだとGeminiがコードを省略したり混乱したりする。
- **解決**: `header`, `guidance`, `faq`, `footer` の4つのHTML部品に分割。
- **結果**: Geminiの出力が安定し、大野さんは「必要な部品だけ」を修正すれば良くなった。

### フェーズ2：プレビューサーバーの構築（試行錯誤）
- **当初の案**: Google Apps Script (GAS) の Web App 機能で Drive 上のファイルを直接レンダリングする。
- **失敗**: 組織のドメインポリシーにより、Web App の外部公開が厳しく制限されており、プレビュー確認が困難。
- **転換**: 動作実績のある **Cloudflare Pages** をプレビュー用として採用。

### フェーズ3：デプロイの自動化（試行錯誤）
- **課題**: `wrangler` コマンドを手動で叩く必要があり、大野さん一人では完結しない。
- **試行1**: GAS から Cloudflare API を直接叩いてファイルをアップロード。
- **失敗**: Cloudflare の Direct Upload API がハッシュ値（Manifest）を要求するため、GAS での実装が複雑化。
- **成功**: **GitHub Actions を「デプロイ職人」として活用。**
  - GAS (Drive変更検知) -> GitHub Repository Dispatch -> GitHub Actions (ビルド&デプロイ) -> Cloudflare Pages
- **結果**: 大野さんは Drive を上書きするだけで、数分後にサイトが更新される環境が完成。

## 3. 成功したポイント
- **SSOT (Single Source of Truth) の維持**: GitHub ではなく Google Drive を正本と定義。
- **GitHub のインフラ化**: GitHub は裏側の自動化ツールとしてのみ使用し、ユーザーには意識させない。
- **セキュリティ**: APIトークンを GAS の Script Properties や GitHub Secrets に隠蔽し、安全性を確保。

## 4. うまくいかなかったこと・課題
- **GAS Web App**: 企業のドメイン管理者による制限は、GAS 単体では突破できない壁であった。
- **ブラウザツールの不安定さ**: 大規模な管理画面（Cloudflareなど）の操作は、ブラウザエージェントがタイムアウトしやすく、手動でのトークン取得が必要になった。
